# Promotion Rubric

Last updated: 2026-06-29

## Purpose

This rubric converts the one-year roadmap into a Google-style public-practices growth ladder for an AI agent architect track.

This is an adaptation, not a claim about Google private internal ladders.

## Level Model

### L3-like: Guided Engineer

Expected signals:

- Can implement scoped tasks with clear guidance
- Understands local module boundaries
- Writes basic tests and docs when asked
- Needs help with difficult trade-offs and failure modeling

Promotion evidence to next level:

- finishes monthly scope consistently
- explains code and interfaces clearly
- begins to anticipate edge cases without being prompted

### L4-like: Independent Engineer

Expected signals:

- Owns small to medium features end to end
- Writes usable RFCs and interface contracts
- Makes reasonable trade-offs independently
- Produces dependable tests and reviewable changes

Promotion evidence to next level:

- improves Atlas beyond assigned tasks
- defends design decisions with evidence
- builds reliable operational habits

### L5-like: Senior Engineer

Expected signals:

- Owns cross-module designs and integration quality
- Builds systems with reliability, observability, and review discipline
- Raises code health across the system
- Coaches others through docs, reviews, and system explanation

Promotion evidence to next level:

- shapes roadmap and interfaces, not just implementation
- demonstrates production-minded architecture judgment
- improves team or project effectiveness through reusable standards

### L6-like: Staff-Ready Architect

Expected signals:

- Designs platform-level boundaries and evolution paths
- Connects product goals, engineering constraints, and operational reality
- Makes hard trade-offs under ambiguity without losing rigor
- Builds architecture others can extend safely

Readiness evidence:

- quarter-level Atlas reviews show durable system improvement
- benchmark, reliability, and governance evidence are integrated
- architecture defense is clear, credible, and complete

## Evaluation Dimensions

### Results

- weak: scope often slips or closes without evidence
- meets: scope closes with evidence and acceptable quality
- strong: scope closes with leverage and reusable outputs

### Technical Judgment

- weak: decisions are preference-driven or inconsistent
- meets: decisions fit constraints and are explained clearly
- strong: decisions simplify future change and reduce system risk

### Code Health

- weak: changes are fragile or hard to review
- meets: changes are readable, testable, and maintainable
- strong: changes make the whole codebase easier to evolve

### Reliability

- weak: tests and failure thinking are late or shallow
- meets: critical paths are validated and recovery is considered
- strong: observability, rollback, and resilience shape the design early

### Communication

- weak: docs are stale and system explanation is unclear
- meets: docs, RFCs, and runbooks are current enough for reuse
- strong: others can quickly onboard and operate from the learner's artifacts

### Community and Learning

- weak: feedback is defensive or absent
- meets: feedback loops are active and visible
- strong: the learner improves standards for others through review and coaching

## Promotion Decision Rules

- No promotion with missing core artifacts.
- No promotion if code health or reliability is persistently weak.
- Promotion requires evidence from Atlas, not only interview-style answers.
- Promotion readiness must be visible in at least one quarter of consistent behavior, not one strong week.

## Year-End Interpretation

- End of Semester 1 target: strong L3-like / emerging L4-like
- End of Semester 2 target: stable L4-like
- End of Semester 3 target: strong L4-like / emerging L5-like
- End of Semester 4 target: stable L5-like with selected L6-like signals
