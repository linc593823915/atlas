# Mentor and Reviewer Handbook

Last updated: 2026-06-29

## Purpose

This handbook explains how mentors and reviewers should use the program documents consistently.

## Review Philosophy

- review for evidence, not enthusiasm
- review for code health, not taste
- review for system movement, not activity volume
- review for growth trajectory, not one isolated strong week

## Monthly Review Workflow

1. Read the monthly scorecard.
2. Read the monthly review packet.
3. Inspect the linked Atlas backlog and issue snapshot.
4. Review one or two representative issue cards.
5. Record strengths, gaps, and next-month focus.

## Quarterly Review Workflow

1. Read the quarterly KPI sheet.
2. Read the quarter replay pack.
3. Inspect the strongest and weakest evidence artifacts.
4. Run the mock panel or equivalent interview loop.
5. Update readiness and promotion signal.

## What Good Review Looks Like

- specific
- evidence-linked
- trade-off aware
- forward-looking

## What Bad Review Looks Like

- generic praise
- taste-based criticism
- no link to evidence
- no distinction between local issue and systemic issue

## Calibration Questions

1. What changed in Atlas that proves growth?
2. Which dimension is improving fastest?
3. Which dimension is underdeveloped despite visible activity?
4. What should be de-scoped or re-scoped next?
5. Is the learner’s self-assessment realistic?

## Review Outputs

- one summary judgment
- one strongest signal
- one most urgent gap
- one next-step recommendation
