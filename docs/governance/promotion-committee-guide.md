# Promotion Committee Guide

Last updated: 2026-06-29

## Purpose

This guide explains how to decide promotion or promotion-readiness using the program artifacts.

## Committee Inputs

- Promotion Rubric
- Annual Readiness Matrix
- Monthly Scorecards
- Quarterly KPI Sheets
- Quarter Replay Packs
- Atlas architecture and backlog evidence

## Review Principle

- promote on sustained evidence, not one strong presentation
- promote on Atlas movement, not only clean documentation
- promote on judgment and reliability, not only delivery speed

## Required Questions

1. Is the learner independently dependable?
2. Is technical judgment visible across multiple months?
3. Is code health improving because of the learner?
4. Is reliability treated as design, not patchwork?
5. Is communication reusable by other engineers?

## Strong Promotion Signals

- consistent month-by-month execution
- credible quarterly architecture reasoning
- visible system-wide improvement in Atlas
- review and documentation quality that helps others

## Weak Promotion Signals

- isolated strong week with no sustained follow-through
- strong answers without artifacts
- closed issues without strong evidence
- good execution but shallow architectural reasoning

## Committee Output

- ready
- nearly ready
- not ready
- conditions for re-review
