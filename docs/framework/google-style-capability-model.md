# Google-Style Capability Model

Last updated: 2026-06-29

## Scope

This framework is an adaptation of Google’s public engineering, management, SRE, testing, and learning materials for an AI agent architect individual-contributor track.

It is not a claim about Google’s private internal ladders.

## Public Signals Used

- Google re:Work OKRs emphasize ambitious objectives, measurable key results, and regular grading.
- Google re:Work manager research highlights Deliver Results, Develop People, and Build Community.
- Google’s team effectiveness research emphasizes psychological safety, dependability, structure and clarity, meaning, and impact.
- Google Engineering Practices focuses on code health, small self-contained changes, review quality, and documentation.
- Google SRE material emphasizes SLOs, release engineering, automation, toil reduction, alerting, incident learning, and reliability.
- Google Testing Blog reinforces strong feedback loops and testing discipline as part of engineering culture.

## Adapted Dimensions for an AI Agent Architect

### 1. Deliver Results

- Turn objectives into shipped artifacts and measurable progress.
- Use explicit goals and key results rather than vague effort tracking.

### 2. Technical Judgment

- Pick the right abstraction, system boundary, and trade-off.
- Prefer facts, evidence, and constraints over taste.

### 3. Code Health and Simplicity

- Improve maintainability over time.
- Favor small changes, clear names, simple flow, and reviewable deltas.

### 4. Reliability and Testing

- Treat failure modes, tests, rollback, and observability as first-class design concerns.
- Build for repeatable releases rather than heroic recovery.

### 5. Communication and Documentation

- Explain intent, interfaces, and operations clearly.
- Keep RFCs, reviews, runbooks, and design notes current.

### 6. Build Community and Learning

- Seek and give feedback with respect.
- Develop peer coaching habits and contribute to team effectiveness.

## Assessment Cadence

- Daily: execution discipline and evidence capture.
- Weekly: shipped artifact plus technical explanation.
- Monthly: capability review against OKRs and interview readiness.
- Quarterly: calibration across results, judgment, reliability, communication, and growth.
- Year-end: architecture defense using Atlas as the main evidence base.

## Score Interpretation

- Not Yet: missing core artifacts, weak explanations, or unreliable execution.
- Meets: shipped expected artifacts with solid reasoning and acceptable quality.
- Strong: shipped the work, improved the system beyond the assignment, and showed architect-level judgment.
