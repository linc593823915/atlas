# Burnup and Burndown

Last updated: 2026-06-29

## Purpose

Burnup and burndown views make progress visible without hiding scope change.

## Monthly Burndown

Track:

- planned issues remaining by week
- blocked issues remaining by week
- done issues by week

Use when:

- the month has a fixed backlog
- you want to see whether issue closure matches the calendar

## Monthly Burnup

Track:

- total planned issues
- total done issues
- total evidence-linked issues

Use when:

- scope changes during the month
- you want to avoid pretending the original plan stayed constant

## Quarterly Burnup

Track:

- months completed
- quarter milestones completed
- quarter evidence packs completed

Use when:

- checking whether the quarter is on track
- explaining pace to a mentor or reviewer

## Data Sources

- issue status board
- issue cards
- review packets
- evidence registry

## Interpretation Rules

- If done issues rise but evidence-linked issues do not, progress is weak.
- If monthly scope grows but burnup stays flat, planning is unrealistic.
- If blocked count rises for multiple weeks, review pacing or dependency handling is failing.
