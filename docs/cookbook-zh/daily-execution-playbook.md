# 日级执行动作手册

最后更新：2026-06-30

## 这份手册解决什么问题

day 文档已经告诉你每天要读什么、写什么、做什么。

但很多句子仍然像动作提示：

- `写复盘、memory 草稿和下一步建议`
- `打通第一条 end-to-end 主路径`
- `归档本周最终证据包`
- `补齐必要的扩展路径、回退逻辑或真实场景下的异常处理`

如果只看这些句子，学习者仍然可能不知道：

1. 这个动作最少要做到什么程度
2. 什么证据能证明它完成了
3. 做过头和做不够分别是什么样

这份手册就是把日级动作翻译成可执行标准。

## 写作动作

<a id="write-week-problem-statement"></a>
### `写一份本周问题说明`

- 最低要写清：
  - 本周问题是什么
  - 为什么现在要处理
  - 本周目标
  - 明确非目标
  - 成功标准
- 合格证据：
  - 另一个人读完后能判断本周该做什么、不该做什么
- 常见错误：
  - 把问题说明写成愿望清单
  - 只写功能，不写约束

<a id="write-reviewable-interface-note"></a>
### `写可评审的 RFC / interface note`

- 最低要写清：
  - 输入
  - 输出
  - owner
  - 错误语义
  - 版本或演进假设
  - 未决问题
- 合格证据：
  - reviewer 能指出一个明确边界，而不是只能评价“想法不错”
- 常见错误：
  - 只写 happy path
  - 把实现细节伪装成 interface

<a id="write-minimal-main-path-note"></a>
### `写出最小主路径的实现说明`

- 最低要写清：
  - 输入从哪里来
  - 控制流经过哪些层
  - 输出到哪里去
  - 哪个不变量必须成立
  - 哪个失败路径暂时不覆盖
- 合格证据：
  - 说明能和代码、测试或命令输出互相对上
- 常见错误：
  - 只写“做了什么”，不写“为什么这条路径成立”

<a id="write-boundary-fallback-list"></a>
### `写边界条件、异常处理和 fallback 清单`

- 最低要写清：
  - 主要边界条件
  - 失败类型
  - retry / fallback / fail closed 策略
  - operator 或 reviewer 能看到的证据
- 合格证据：
  - 至少一条失败路径能被测试、drill 或手工复现
- 常见错误：
  - 把 fallback 写成“失败了再想办法”
  - 没有说明 fallback 后输出是否仍满足 contract

<a id="write-validation-matrix"></a>
### `写测试样例、drill 方案或验证矩阵`

- 最低要写清：
  - 要证明的不变量
  - 正向样本
  - 失败样本
  - 运行方式
  - 通过 / 失败判断
- 合格证据：
  - 别人能按矩阵复跑，并得到同一种判断
- 常见错误：
  - 只列测试名，不写它保护什么

<a id="write-review-memory-next"></a>
### `写复盘、memory 草稿和下一步建议`

- 最低要写清：
  - 本周最稳定的结论
  - 最强证据
  - 最大残余风险
  - 下周第一步
  - 哪条经验值得写入 memory
- 合格证据：
  - 下周开始时能直接根据这份复盘开工
- 常见错误：
  - 写成学习心得
  - 没有证据链接
  - 下一步太大，无法执行

<a id="write-handover-doc"></a>
### `写运行说明、review note 或 operator 文档`

- 最低要写清：
  - 当前能力怎么运行
  - 关键设计取舍
  - 已知限制
  - 常见失败怎么排查
  - reviewer 最该看的证据
- 合格证据：
  - 不熟悉这周上下文的人能接手继续改
- 常见错误：
  - 只解释代码，不解释操作和风险

## 动手动作

<a id="prepare-task-evidence-boundary"></a>
### `整理任务清单、证据目录和最小落地边界`

- 最低要做：
  - 把任务拆成 3-5 个可执行项
  - 标出每项完成证据
  - 写清今天不做什么
  - 找到对应代码、文档或 issue 入口
- 合格证据：
  - 今天结束时能判断哪些项完成，哪些项推迟
- 常见错误：
  - 任务清单只有动词，没有验收证据

<a id="land-structure-boundary-diagram"></a>
### `把最小结构图、包边界、状态图或协议草图真正落到仓库笔记里`

- 最低要做：
  - 选择一种图或表
  - 写清节点 / 状态 / 字段
  - 写清调用或迁移方向
  - 标出 owner
- 合格证据：
  - 这张图能解释一个真实代码或协议路径
- 常见错误：
  - 画了漂亮图，但不能指导实现或 review

<a id="open-first-e2e-path"></a>
### `打通第一条 end-to-end 主路径`

- 最低要做：
  - 选择最小输入
  - 从入口跑到可观察输出
  - 记录命令或测试
  - 记录成功信号
  - 标出未覆盖失败路径
- 合格证据：
  - 另一个人能复跑同一条路径
- 常见错误：
  - 只跑局部函数，却声称 e2e 已经成立

<a id="run-validation-or-smoke"></a>
### `运行测试、drill、smoke check 或测量`

- 最低要做：
  - 写下运行命令
  - 说明验证目标
  - 保存输出摘要
  - 解释失败或跳过
- 合格证据：
  - 结果能支撑今天的判断，而不是只证明命令跑过
- 常见错误：
  - 把一次本机成功当成稳定证据

<a id="add-realistic-fallback-path"></a>
### `补齐必要的扩展路径、回退逻辑或真实场景下的异常处理`

- 最低要做：
  - 选择一个真实风险
  - 补一条扩展或失败路径
  - 明确何时触发
  - 给出观测信号
  - 不破坏主路径 contract
- 合格证据：
  - happy path 和失败路径都能解释
- 常见错误：
  - 加了复杂度，但没有对应风险

<a id="handover-cleanup"></a>
### `清理命名、结构、注释和文档，把本周成果整理成可交接资产`

- 最低要做：
  - 清掉误导命名
  - 删除或标注临时假设
  - 补最小运行说明
  - 把证据链接放到 review 或周报里
- 合格证据：
  - 接手者不用翻聊天记录也能继续
- 常见错误：
  - 只做格式化，不处理真正的理解成本

<a id="archive-week-evidence-pack"></a>
### `归档本周最终证据包`

- 最低要做：
  - 列出本周交付物
  - 列出最强证据
  - 列出失败或未覆盖项
  - 写下一条可复用结论
  - 指向下周入口
- 合格证据：
  - 月度 review 能直接引用这份证据包
- 常见错误：
  - 归档成文件列表，没有判断

## 最小交付动作

<a id="deliver-problem-statement"></a>
### `一份范围清晰的问题说明`

- 最低要包含：
  - 问题
  - 范围
  - 非目标
  - 成功标准
  - 最小证据

<a id="deliver-reviewable-contract"></a>
### `一份可被 review 的契约草稿`

- 最低要包含：
  - 输入输出
  - owner
  - 错误语义
  - 版本假设
  - reviewer 需要确认的问题

<a id="deliver-runnable-path"></a>
### `一条可运行或可演示的核心路径`

- 最低要包含：
  - 运行入口
  - 输入样本
  - 输出或成功信号
  - 复跑命令
  - 未覆盖风险

<a id="deliver-validation-result"></a>
### `验证结果、失败清单或下一步修复方向`

- 最低要包含：
  - 验证目标
  - 运行结果
  - 失败项
  - 是否阻断
  - 下一步

<a id="deliver-handover-review-note"></a>
### `一份别人能接手的文档资产和 review note`

- 最低要包含：
  - 当前能力说明
  - 关键取舍
  - 证据
  - 已知限制
  - 下一步 owner

<a id="deliver-retro-evidence-pack"></a>
### `一份复盘记录、一组证据包，以及至少一条可复用结论`

- 最低要包含：
  - 本周目标
  - 实际完成
  - 最强证据
  - 最大残余风险
  - 可复用结论

<a id="deliver-extended-usable-path"></a>
### `一条更完整的可用路径，并附上边界清单`

- 最低要包含：
  - 主路径
  - 至少一条失败或扩展路径
  - 边界清单
  - 验证方式
  - 不破坏的 contract

## 高频判断句

<a id="criterion-target-constraints-risk"></a>
### `至少覆盖目标、约束、风险、非目标和成功标准`

- 这句话的意思是：
  - 先把工作边界冻结，再开始做
  - 目标说明要做什么
  - 非目标说明今天不做什么
  - 风险说明最可能失败在哪里
  - 成功标准说明何时能停
- 自检：
  - 如果删掉其中任何一项，是否会导致范围变模糊

<a id="criterion-input-output-owner-evolution"></a>
### `至少明确输入、输出、错误语义、owner、未决问题和版本/演进假设`

- 这句话的意思是：
  - interface note 不能只描述 happy path
  - owner 决定谁负责修改和解释
  - 错误语义决定调用方如何处理失败
  - 演进假设决定下一版怎么不打断别人
- 自检：
  - 另一个调用方能不能只看这份说明就正确接入

<a id="criterion-input-output-invariant"></a>
### `明确输入、输出和关键不变量`

- 这句话的意思是：
  - 输入说明主路径从哪里开始
  - 输出说明主路径到哪里算结束
  - 不变量说明中间不能被破坏的规则
- 自检：
  - 如果这条不变量被破坏，测试或 reviewer 能否发现

<a id="criterion-proof-scope"></a>
### `明确今天要证明什么、不证明什么`

- 这句话的意思是：
  - 今天的验证必须有边界
  - 不证明的部分要显式留下，避免被误当成完成
- 自检：
  - 读者能否分清“已经证明”和“只是还没测”

<a id="criterion-risk-choice-retro"></a>
### `明确最大的收获、最大的风险和最值得保留的做法`

- 这句话的意思是：
  - 复盘要服务下一轮决策
  - 收获必须能转成做法
  - 风险必须能转成下一步
- 自检：
  - 下周开始时，这段复盘能否直接指导第一步

<a id="criterion-tradeoff-risk-limit"></a>
### `把设计取舍、风险边界和已知限制明确写出来`

- 这句话的意思是：
  - 交接文档要说明为什么这样做
  - 风险边界说明哪里可能坏
  - 已知限制说明不要误用到什么场景
- 自检：
  - 接手者能否知道哪些地方不能随便改

<a id="criterion-no-extra-complexity"></a>
### `不顺手扩展额外复杂度`

- 这句话的意思是：
  - 今天只解决当前目标需要的最小问题
  - 额外抽象、额外功能、额外依赖都要有证据理由
- 自检：
  - 如果删掉这部分复杂度，今天的交付是否仍然成立

<a id="criterion-verifiable-not-slogan"></a>
### `写成可验证结果，而不是口号`

- 这句话的意思是：
  - 不能只写“支持 Docker”“完成配置中心”
  - 要写命令、路径、输出、失败信号或证据链接
- 自检：
  - reviewer 能否根据这句话找到一条可检查证据

<a id="criterion-evidence-conclusion"></a>
### `变成有证据的结论`

- 这句话的意思是：
  - 结论必须绑定测试、命令、trace、review 或文档证据
  - 证据要能说明为什么这个结论可信
- 自检：
  - 如果别人不相信你的结论，你能否指向一条证据

<a id="criterion-week-standard-verifiable"></a>
### `让本周至少一项交付标准开始可验证`

- 这句话的意思是：
  - 今天不一定完成整周目标
  - 但至少要让一个交付标准从“计划”变成“可被检查”
- 自检：
  - 本周周文档里的哪一条标准被今天推进了

<a id="criterion-reusable-material"></a>
### `为下周或下个月留下可直接复用的材料`

- 这句话的意思是：
  - 归档要留下判断、证据和入口
  - 不能只留下文件列表
- 自检：
  - 下个月 review 是否能直接引用这份材料

<a id="criterion-supporting-evidence-list"></a>
### `至少能支撑`

- 这句话的意思是：
  - 最小交付不是一个孤立文件
  - 它要能支撑 day 文档列出的证据目标
- 自检：
  - 每个证据目标是否都能找到对应产物或验证记录

## 最后一句

日级动作的标准不是“今天忙过”。

标准是：

今天留下的东西，明天、下周、月度 review 或另一个接手者是否还能用。
