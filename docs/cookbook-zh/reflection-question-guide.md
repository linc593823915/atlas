# 延伸追问回答指南

最后更新：2026-06-30

## 这份指南解决什么问题

day 文档里的 `延伸追问` 不是课后闲聊。

它们的目标是训练你把当天学习转成工程判断：

- 能不能解释边界
- 能不能指出证据
- 能不能识别风险
- 能不能做下一步取舍

如果你只回答“我觉得”“应该可以”“后面再看”，说明追问没有发挥作用。

## 回答通用格式

每个追问至少按四句话回答：

1. 我的判断是什么
2. 我依据什么证据
3. 这个判断的风险或边界在哪里
4. 下一步最小动作是什么

## 问题范围类

<a id="question-system-quality"></a>
### `为什么这个主题影响的是系统质量，而不只是任务完成？`

- 回答重点：
  - 它影响哪条长期边界
  - 如果只当任务完成，会留下什么风险
  - 哪个后续主题会被它影响
- 不合格回答：
  - “因为它很重要”

<a id="question-scope-half"></a>
### `如果把本周范围砍掉一半，我会保留哪条主路径？`

- 回答重点：
  - 哪条路径最能证明本周目标
  - 哪些内容可以推迟
  - 推迟后风险是否可控
- 不合格回答：
  - “都很重要，最好都做”

<a id="question-senior-signal"></a>
### `资深工程师在今天的阅读材料里最先会盯住什么信号？`

- 回答重点：
  - 盯住的不是名词，而是风险信号
  - 说明这个信号对应的系统边界
  - 说明如何在 Atlas 中验证
- 不合格回答：
  - “会看架构”

## 接口与边界类

<a id="question-public-interface"></a>
### `哪些能力应该成为公开接口，哪些细节应该被隐藏？`

- 回答重点：
  - 公开接口要稳定、可消费、可测试
  - 隐藏细节可以变化，不应该被调用方依赖
  - 至少举一个字段、函数、capability 或文档例子
- 不合格回答：
  - “常用的公开，不常用的隐藏”

<a id="question-failure-mode"></a>
### `这份边界设计里最容易漏掉的失败模式是什么？`

- 回答重点：
  - 找一个真实失败路径
  - 说明它为什么容易漏
  - 说明测试、drill 或 review 怎么暴露它
- 不合格回答：
  - “异常情况要注意”

<a id="question-speed-quality-tradeoff"></a>
### `这里的速度与设计质量冲突时，我会如何取舍？`

- 回答重点：
  - 哪些边界不能牺牲
  - 哪些实现细节可以先简化
  - 简化后要留下什么补债记录
- 不合格回答：
  - “看情况”

## 测试与证据类

<a id="question-happy-path-missing-bug"></a>
### `如果我只写 happy path，最可能漏掉什么 bug？`

- 回答重点：
  - 具体说出一个坏输入、超时、拒绝、重复执行或状态不一致场景
  - 说明它如何影响调用方或 operator
  - 说明最小负路径测试
- 不合格回答：
  - “会漏异常”

<a id="question-required-vs-nice-tests"></a>
### `这里哪些测试是必须的，哪些只是锦上添花？`

- 回答重点：
  - 必须测试保护 contract 或不变量
  - 锦上添花测试只提高覆盖率或便利性
  - 用风险排序，而不是用实现难度排序
- 不合格回答：
  - “能写都写”

<a id="question-safe-to-continue-evidence"></a>
### `我会拿什么证据证明下周继续改这里仍然安全？`

- 回答重点：
  - 指向测试、review note、trace、drill 或文档证据
  - 说明证据覆盖哪条边界
  - 说明证据没有覆盖什么
- 不合格回答：
  - “目前没问题”

<a id="question-implementation-invariant"></a>
### `如果这条实现是正确的，哪个不变量必须始终成立？`

- 回答重点：
  - 不变量要能被测试或 review
  - 不变量要和业务结果或系统边界有关
  - 说明违反后会发生什么
- 不合格回答：
  - “代码不报错”

<a id="question-control-flow-explain"></a>
### `把今天打通的控制流或数据流完整讲一遍。`

- 回答重点：
  - 从入口开始
  - 经过每个 owner
  - 说明状态或数据在哪里变化
  - 到输出或失败路径结束
- 不合格回答：
  - “调用 A，然后调用 B”

<a id="question-future-abstraction"></a>
### `我今天选中的哪个抽象是在为未来变更留余地？`

- 回答重点：
  - 抽象必须对应真实变化压力
  - 说明它隐藏了什么变化
  - 说明它今天没有过度设计的证据
- 不合格回答：
  - “以后可能会用”

## Review 与复盘类

<a id="question-week-changing-judgment"></a>
### `这周哪条结论真正改变了我的工程判断？`

- 回答重点：
  - 写一个从旧判断到新判断的变化
  - 指出改变判断的证据
  - 说明以后会怎么做不同
- 不合格回答：
  - “学到了很多”

<a id="question-interview-evidence"></a>
### `如果拿这周产物去做一次面试或评审，哪条证据最能证明成熟度？`

- 回答重点：
  - 选一条最强证据，而不是列清单
  - 说明它证明了什么能力
  - 说明它的局限
- 不合格回答：
  - “文档和代码都可以”

<a id="question-next-debt"></a>
### `下次再做同类主题时，哪块债务必须先处理？`

- 回答重点：
  - 债务必须具体到边界、测试、文档或运行路径
  - 说明不处理会影响什么
  - 给出第一步动作
- 不合格回答：
  - “还要继续优化”

<a id="question-code-health-review"></a>
### `如果按代码健康标准评审，这项工作最该被追问的点是什么？`

- 回答重点：
  - reviewer 会追 owner、边界、错误语义、测试或可维护性
  - 说明追问背后的风险
  - 说明你已有或缺少的证据
- 不合格回答：
  - “代码质量还行”

<a id="question-missing-docs"></a>
### `在真正算完成之前，还缺哪类文档或运行说明？`

- 回答重点：
  - 区分开发者文档、operator 文档、review note、runbook
  - 说明缺文档会让谁无法行动
  - 说明最小补法
- 不合格回答：
  - “补 README”

<a id="question-review-feedback"></a>
### `如果这是同事的改动，我会给出哪条最关键的 review feedback？`

- 回答重点：
  - 反馈要指向风险和证据
  - 不要只给风格建议
  - 说明是否阻断合并
- 不合格回答：
  - “写得不错，可以再完善”

## 扩展与规模类

<a id="question-scale-breakpoint"></a>
### `如果规模、调用量或团队人数增长十倍，这里会先在哪断？`

- 回答重点：
  - 指出一个容量、协作、接口或治理断点
  - 说明十倍增长如何触发它
  - 说明应该提前留下什么观测或边界
- 不合格回答：
  - “性能可能不够”

<a id="question-first-boundary-failure"></a>
### `哪条边界最可能最先出故障，我打算如何探测它？`

- 回答重点：
  - 选择一条具体边界
  - 说明故障信号
  - 说明探测方式
- 不合格回答：
  - “多测一下”

<a id="question-simplest-usable-version"></a>
### `满足今天需求的最简单可用版本到底是什么？`

- 回答重点：
  - 最简单版本要仍然满足 contract
  - 明确砍掉哪些非必要能力
  - 说明保留哪些验证
- 不合格回答：
  - “先做 MVP”

## 年度答辩类

<a id="question-platform-evidence"></a>
### `你会拿哪组证据证明 Atlas 已经从“学习工程”进化成“可演进的平台雏形”？`

- 回答重点：
  - 选择架构、运行、质量、治理、性能中的几组强证据
  - 说明它们如何互相支撑
  - 说明仍然不像成熟平台的地方

<a id="question-defend-design-decision"></a>
### `如果现在做一次完整架构答辩，你最有把握 defending 的设计决策是什么？`

- 回答重点：
  - 选择一个具体决策
  - 说明背景、取舍、证据和残余风险
  - 说明为什么不是另一个方案

<a id="question-next-specialization"></a>
### `未来一年里，哪块能力最值得继续深挖，为什么不是别的方向？`

- 回答重点：
  - 从当前证据和风险推出方向
  - 说明机会成本
  - 说明下一阶段最小项目

## 最后一句

延伸追问的答案不需要长。

但必须可检查。

一个合格答案应该让 reviewer 看出：

- 你做了判断
- 你知道证据在哪里
- 你知道边界和风险
- 你知道下一步怎么推进
