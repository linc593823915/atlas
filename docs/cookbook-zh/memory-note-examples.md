# Memory 示例

最后更新：2026-06-30

## 这份文档解决什么问题

学习者通常知道：

- 每做完一块要更新 `.memory`
- `.memory` 很重要

但真正落笔时，最常见的情况是：

- 写成流水账
- 写成一句空泛总结
- 写不出未来还能复用的结论

这份文档就是把当前仓库里的 `.memory/issues/*.md` 写法讲透。

## 当前仓库实际在用什么格式

从现有条目看，当前最稳定的格式就是三段：

```md
Decision:
...

Reason:
...

Future:
...
```

这个格式的好处是：

1. `Decision` 强迫你给出结论
2. `Reason` 强迫你说明为什么要这么做
3. `Future` 强迫你留下下一步或残余缺口

## 一条好的 Memory 不该写成什么

### 不好的写法 1：流水账

```md
Decision:
今天改了很多文档。

Reason:
用户让我继续补。

Future:
继续补。
```

问题在于：

- 没有可复用结论
- 没说明具体改了什么能力边界
- 未来任何人都无法从这条 memory 学到判断

### 不好的写法 2：只有态度，没有结构

```md
Decision:
教材现在更好了。

Reason:
因为内容更多了。

Future:
继续优化。
```

问题在于：

- “更好”没有对象
- “内容更多”不是工程理由
- “继续优化”不能指导后续动作

## 示例 1：教材结构层 Memory

### 场景

- 你刚把 month/week/day 从概念骨架推进成可执行学习单

### 更好的写法

```md
Decision:
中文 cookbook 的 month / week / day 学习层补入知识清单、评分标准、面试追问和日入口。

Reason:
原有中文教材虽然已经覆盖一年路径，但学习者面对 month / week / day 时仍然偏空：
- 只有主题，没有知识展开
- 只有步骤，没有验收口径
- 只有概念，没有追问与自检

要让教材真正可用，必须把它从“概念索引”推进成“可执行学习单”。

Future:
专题案例讲解
中文术语表
跨月知识索引
```

### 这条为什么更强

因为它留下的是：

- 具体决策
- 明确问题
- 可接续的后续方向

## 示例 2：执行路径层 Memory

### 场景

- 你刚把周层补成 `本周最小演示场景` 和 `本周工作顺序`

### 更好的写法

```md
Decision:
中文 cookbook 的 week 层增加“本周最小演示场景”和“本周工作顺序”。

Reason:
即使 week 层已经有：
- 知识清单
- Atlas 观察路径
- 配套案例

学习者仍然可能不知道：
- 这一周结束时到底该怎么演示
- 这一周按什么顺序推进 RFC、实现、验证和文档

把这两层写进 week 正文后，week 层才真正从“主题页”变成“执行页”。

Future:
把 week 工作顺序继续和具体 day 产物互链
补主题级源码讲解
补 review checklist 模板
```

### 这条为什么更强

因为它不是在写“我新增了两段文字”，而是在写：

- 这一改动在学习路径里扮演什么作用

## 示例 3：交付物方法论 Memory

### 场景

- 你刚写完 `交付物写作指南`

### 更好的写法

```md
Decision:
中文 cookbook 增加“交付物写作指南”，系统化说明 RFC、Interface、Implementation、Unit Test、Review、Benchmark、Documentation、Memory 的最小写法。

Reason:
教材已经告诉学习者：
- 学什么
- 做什么
- 看什么

但仍然缺少：
- 这些交付物到底怎么写
- 怎样判断自己写的产物够不够

如果没有这一层，学习者很容易知道流程名词，却交不出合格产物。

Future:
把写作指南继续和月 / 周实战清单互链
补更多产物示例
补 issue / evidence entry 示例
```

### 这条为什么更强

因为它沉淀的是一套“以后也会反复用到”的判断，而不是某次局部改动。

## 示例 4：治理与外部阅读层 Memory

### 场景

- 你刚把外部阅读提示从纯文本收成了解读文档

### 更好的写法

```md
Decision:
中文 cookbook 增加“外部阅读提示解读”，把高频外部阅读提示从纯文本名称翻译成明确阅读目标。

Reason:
day 文档里长期存在一类提示：
- OpenAI production best practices
- Kubernetes overview and OpenTelemetry overview
- NIST AI RMF GenAI profile

如果只保留名称，学习者往往不知道：
- 为什么现在要读它
- 读完后该把什么判断带回 Atlas

把它们收口成一份文档后，外部阅读才真正变成学习路径的一部分。

Future:
继续吃掉剩余零散外部提示
把更多外部提示接成稳定链接
减少 day 文档中的抽象英文短语
```

## 写 Memory 时最该问的三个问题

1. 这条结论以后还会不会复用？
2. `Reason` 写的是工程原因，还是只是在复述动作？
3. `Future` 是否真的能指导后续推进？

## 一条合格的 Memory 最低标准

至少要满足：

- `Decision` 不是空泛判断
- `Reason` 里能看出真实问题或约束
- `Future` 至少留下 1-3 个可推进方向

## 最后一句

Memory 的目标不是存档情绪，也不是证明你做了很多事。

它的目标是：

把一条以后还会影响同类工作的判断，用最短的方式留下来。
