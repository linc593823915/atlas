# 全年总览

最后更新：2026-06-30

## 全年地图

这一年被分成四个学期。

每个学期都有不同的目标：

### 第一学期：基础与第一个 Agent

你会建立后端基础、分布式执行基础，并做出第一个能工作的单 Agent 服务。

结果是：

- 不再只是会调模型
- 开始具备“工程化包装模型能力”的基础

### 第二学期：接口、协议与运行时

你会从“单条功能链路”升级到“可复用系统接口”：

- Tool Gateway
- MCP
- Multi-Agent Runtime
- Approval Flow

结果是：

- 开始具备平台级思维
- 不再只是做一个功能，而是设计可扩展边界

### 第三学期：状态、评估与治理

你会让系统真正进入企业级复杂度：

- Durable Workflow
- Memory Boundary
- Evals
- Traces
- Governance

结果是：

- 你开始理解“为什么很多 Agent Demo 一上线就崩”
- 并且知道如何用系统方法把它们补上

### 第四学期：生产、性能与平台整合

你会把 Atlas 推到更接近内部平台的形态：

- Kubernetes
- SLO / Rollback
- Benchmark / Capacity
- Platform Integration

结果是：

- 你不只是能设计一条 Agent 流程
- 而是能讨论一个 Agent 平台是否可运营

## 四个学期的学习逻辑

顺序不是随便排的。

它遵循真实工程成长顺序：

1. 先会写服务
2. 再会设计接口
3. 再会治理复杂性
4. 最后才谈平台与规模

很多人失败，是因为顺序反了：

- 一上来就讨论多 Agent
- 一上来就谈自治规划
- 一上来就谈记忆系统

但底层连：

- 配置
- 日志
- 测试
- 失败恢复

都没有处理好。

## 全年的真正主线

这不是“52 个独立小练习”。

真正主线只有一条：

`Atlas 持续演进`

所以你每学完一块，都要问自己：

- 这块能力有没有进入 Atlas？
- 它是不是以后还能复用？
- 它有没有让系统更接近平台？

## 建议的进度判断

### 到第 3 个月末

你应该至少能独立做出一个：

- 有配置
- 有日志
- 有工具调用
- 有结构化输出

的单 Agent 服务。

### 到第 6 个月末

你应该能解释：

- Tool 为什么要有统一入口
- MCP 为什么重要
- Multi-Agent 为什么不能只靠 Prompt 粘起来

### 到第 9 个月末

你应该已经理解：

- 为什么 Agent 必须有评估
- 为什么必须有 Tracing
- 为什么必须做安全和治理

### 到第 12 个月末

你应该具备：

- 平台级架构表达能力
- 企业级系统权衡能力
- 以 Atlas 为证据的工程说服力
